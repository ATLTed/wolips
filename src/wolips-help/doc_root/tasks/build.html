How-to use documentation:

  1. make sure that WOLips is properly installed and the build.properties file exists
  2. create Java Projects for the WebObjects Projects
  3. add a filter to the Package Explorer filtering out things called build (and eg. WebComponents.*, for theCMSFramework)
  4. add a WebObjects Classpath container to the buildpath (Project Properties / Build Path / Libraries / Advanced) and choose all the WebObjects Frameworks you wish to use. (This step can be omitted on applications that use a common framework if that framework exports the WebObjects classpath and if the application doesn't need additional WO frameworks.)
  5. add all the .jar-files that are needed by the project (as "Internal Jar" if they are in CVS, via add Variable if they are not).
  6. fix all the compilation problems (like wrong package names in classes)
  7. edit the Project Properties and check the "Is a WebObjects Project" checkbox (also check "Is a Framework" for frameworks).
  8. create a launch configuration using the Run... menu
  9. change the runtime classpath and the source path to not be the default paths (uncheck both checkboxes)
  10. (do this only after unchecking both checkboxes in the step above:) change the Runtime Classpath by adding the build/<projectname>.woa/Contents/Resources/Java Folder to the classpath before or instead of the Project itself
  11. (do the same for all WO-Frameworks in your Workspace, using the appropriate Resources/Java folders).
  12. add the parameters -Duser.dir=<path to the .woa directory under build> (or better: set this folder as the working directory for the application) and -DWORootDirectory=>$NEXT_ROOT< (replacing >$NEXT_ROOT< with the installation path of your WebObjects installation)
  13. turn on reloading of templates with -WODebuggingEnabled true
  14. turn on rapid-turnaround mode with -NSProjectSearchPath ("../..")
  15. optional: set the port the application listens on with (e.g.) -WOPort 6666
  16. run the application
